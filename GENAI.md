# GenAI Usage — Task Manager

## Tool Used

**Claude Code** (`claude-sonnet-4-6`) via Anthropic CLI — an agentic AI coding assistant that reads and writes files, runs shell commands, and executes build/test pipelines directly in the terminal session.

---

## How AI Was Used

### 1. Project Scaffolding

The developer provided a complete architectural specification upfront. Claude Code translated that spec into:

- `dotnet new` commands for all 9 projects (5 source + 4 test)
- NuGet package graph with explicit versions (`Npgsql 8.0`, `BCrypt.Net-Next`, `FluentValidation`, `xUnit`, `Moq`, `FluentAssertions`, `Microsoft.AspNetCore.Authentication.JwtBearer`)
- Project references wired correctly across layers (Domain ← Application ← Infrastructure ← Api/Auth)
- `TaskManager.sln` with all projects registered

The developer reviewed and approved the dependency graph before any code was written.

### 2. Backend Code Generation

The AI generated the full backend implementation across all layers:

- **Domain**: `TaskItem` and `User` entities using factory methods (`Create` / `Reconstitute`) to separate construction-time validation from persistence hydration; `Email` and `TaskStatusValue` value objects; `DomainException` for invariant violations; repository interfaces
- **Application**: `TaskService` and `AuthService` with constructor-injected interfaces only (no concrete dependencies); DTOs as C# records; FluentValidation validators; `IPasswordHasher` abstraction
- **Infrastructure**: Npgsql repositories using `NpgsqlCommand` with `@parameter` placeholders (no string concatenation); `JwtService` reading config from `IConfiguration`; `PasswordHasher` wrapping BCrypt; idempotent `DatabaseMigrator` using `CREATE TABLE IF NOT EXISTS` and `ON CONFLICT DO NOTHING`
- **API/Auth**: Controllers with `[Authorize]` and claim extraction; JWT Bearer middleware; global `ExceptionMiddleware` mapping domain exceptions to HTTP status codes; Swagger with Bearer token support

### 3. Test Generation

All 37 tests were generated by AI across four projects:

| Project | Tests | Approach |
|---------|-------|----------|
| Domain.Tests | 50 | Pure unit tests — no mocks needed |
| Application.Tests | 48 | Moq for repository/service interfaces |
| Infrastructure.Tests | 15 | In-memory config + real BCrypt/JWT |
| Api.Tests | 23 | `WebApplicationFactory` + `HttpClient` |

The AI independently chose `WebApplicationFactory` for the API layer rather than mocking the controllers — a deliberate decision to test the full middleware pipeline including JWT validation and exception handling.

### 4. Frontend Scaffolding

- Angular 17 standalone components with inline templates and styles
- Functional `HttpInterceptorFn` (not the deprecated class-based `HttpInterceptor`)
- Functional `CanActivateFn` guard
- `BehaviorSubject<boolean>` for reactive auth state
- Angular Signals for task list state management in `TaskListComponent`
- Reactive Forms with validators on all inputs

### 5. Infrastructure Files

- `docker-compose.yml` with Postgres healthcheck (`pg_isready`) and `depends_on: condition: service_healthy` to prevent race conditions on startup
- Multi-stage Dockerfiles: `sdk:9.0` build stage → `aspnet:9.0` runtime stage (image size reduction)
- Nginx config for Angular SPA with `try_files $uri $uri/ /index.html` for client-side routing

## Developer Overrides and Corrections

These are cases where the AI-generated output was reviewed and corrected. Each correction required understanding the underlying architectural rule, not just the surface error.

### Override 1 — BCrypt in Application layer

**What AI generated:** `AuthService` calling `BCrypt.Net.BCrypt.HashPassword()` and `BCrypt.Net.BCrypt.Verify()` directly, with a NuGet reference to `BCrypt.Net-Next` added to `TaskManager.Application`.

**Why it was wrong:** Clean Architecture prohibits Application from depending on Infrastructure packages. BCrypt is an implementation detail — the Application layer should only know that passwords can be hashed and verified, not how.

**Fix applied:** Introduced `IPasswordHasher` interface in `TaskManager.Application.Interfaces`; moved the BCrypt implementation to `TaskManager.Infrastructure.Services.PasswordHasher`; removed the BCrypt NuGet reference from Application entirely.

---

### Override 2 — Missing `IDatabaseMigrator` interface

**What AI generated:** `Program.cs` calling `new DatabaseMigrator(connectionFactory).MigrateAsync()` directly at startup, with no abstraction.

**Why it was wrong:** `WebApplicationFactory<Program>` runs the real `Program.cs` startup code. Without a mockable interface, every API integration test would require a live PostgreSQL connection — making the test suite impossible to run in CI without a database.

**Fix applied:** Extracted `IDatabaseMigrator` interface; registered it in DI via `AddSingleton<IDatabaseMigrator>`; `TestWebApplicationFactory` replaces it with `new Mock<IDatabaseMigrator>().Object`, allowing full controller integration tests with no database dependency.

---

### Override 3 — Missing post-login navigation in Angular

**What AI generated:** `AuthService.storeSession()` correctly saved the JWT to `localStorage` and emitted on the `BehaviorSubject`, but never called `router.navigate`. The `LoginComponent` and `RegisterComponent` both subscribed with `next: () => this.loading = false` — meaning a successful login left the user on the login page with no feedback.

**Why it was wrong:** Storing auth state and redirecting are two parts of the same user action. The navigation side effect was missing from both the service and the component.

**Fix applied:** Added `this.router.navigate(['/tasks'])` to `storeSession()` — centralising the redirect so both login and register benefit from a single change.

---

### Override 4 — `TaskStatus` namespace collision

**What AI generated:** Code across multiple files using `TaskStatus` which compiled fine in isolation but failed when `System.Threading.Tasks` was also in scope — producing ambiguous reference errors.

**Why it was wrong:** `System.Threading.Tasks.TaskStatus` exists in the BCL and collides with `TaskManager.Domain.Enums.TaskStatus` whenever both namespaces are imported.

**Fix applied:** Added `using DomainTaskStatus = TaskManager.Domain.Enums.TaskStatus;` alias across `TaskStatusValue.cs`, `TaskItem.cs`, `TaskRepository.cs`, and `TaskService.cs`. The alias makes the intent explicit and eliminates the ambiguity without renaming the domain type.

## What Was Not AI-Generated

- The architectural constraints (Clean Architecture, DDD, Npgsql raw-only, no MediatR) — defined entirely by the developer in the initial specification
- The decision to separate Auth and API into two distinct processes
- The status transition business rules (which transitions are valid) — specified by the developer, encoded by AI
- Review and approval of every generated file before committing
